### [Ch0-2] 바이브 코딩이란?

2025년 2월, 테슬라와 OpenAI의 공동 창업자인 **안드레이 카파시**가 X(구 트위터)에 이런 글을 올렸습니다. "나는 요즘 '바이브 코딩'을 하고 있다. AI에게 요청하고, 결과물을 받아 실행하고, 에러가 나면 다시 AI에게 붙여넣는다. 그게 전부다."

**[슬라이드의 정의를 가리키며]** 바이브 코딩은 자연어로 AI에게 요청하고, AI가 코드를 생성하면, 개발자는 그것을 검토하고 방향을 잡는 개발 방식입니다. 전통적인 방식처럼 모든 문법을 외우고 직접 타이핑할 필요가 없습니다.

중요한 것은 AI가 생성한 코드를 **이해하고 검토하는 능력**입니다. AI는 훌륭한 조수지만, 방향을 정하는 것은 여러분입니다.

---

### [Ch0-3] 전통적 개발 vs 바이브 코딩

**[왼쪽 전통 방식을 가리키며]** 기존 방식을 볼까요? 스택 오버플로우를 뒤지고, 공식 문서를 읽고, 직접 코드를 작성하고, 에러가 나면 다시 구글링. 이 과정이 무한 반복됩니다. 간단한 API 연동 하나에도 반나절이 걸릴 수 있어요.

**[오른쪽 바이브 코딩을 가리키며]** 바이브 코딩 방식은 다릅니다. "이런 기능이 필요해"라고 자연어로 말하면 AI가 코드를 만들어줍니다. 우리는 그것을 실행해보고, 필요하면 수정을 요청합니다.

속도 차이가 얼마나 날까요? 수자이 프로젝트에서 직접 경험한 결과, 전통 방식 대비 3~5배 빠르게 개발할 수 있었습니다. 이 워크샵이 끝나면 여러분도 똑같이 경험하게 될 겁니다.

### [Ch0-4] 바이브 코딩의 3가지 원칙

바이브 코딩을 잘하려면 3가지 원칙을 기억하세요.

**[첫 번째 원칙]** "AI는 페어 프로그래머다." 옆에 시니어 개발자가 앉아있다고 생각하세요. 하지만 이 개발자는 지치지 않고, 항상 친절하고, 어떤 질문이든 답해줍니다. 그게 AI입니다.

**[두 번째 원칙]** "코드를 이해하고 검토하라." AI가 만들어준 코드를 무조건 복붙하면 안 됩니다. 내가 이해하지 못하는 코드는 내 코드가 아닙니다. 나중에 버그가 생겼을 때 수정도 못 합니다.

**[세 번째 원칙]** "작게 나누어 요청하라." 한 번에 전체 시스템을 만들어달라고 하면 AI도 헤맵니다. 기능 단위로 쪼개서 요청하는 것이 훨씬 좋은 결과를 얻는 방법입니다.

---

### [Ch0-5] AI 도구 선택 가이드

현재 사용할 수 있는 주요 AI 코딩 도구를 소개합니다.

**[Claude를 가리키며]** Claude는 복잡한 코드 설명과 긴 컨텍스트 처리에 강합니다. 설계 문서나 요구사항 작성에 특히 유용합니다. 오늘 우리가 주로 사용할 도구입니다.

**[ChatGPT를 가리키며]** ChatGPT는 가장 범용적인 도구입니다. GPT-4o 이상 버전을 사용하면 코딩 품질이 좋습니다.

**[Cursor를 가리키며]** Cursor는 IDE 자체에 AI가 내장된 도구입니다. 파일 전체를 보면서 수정해주기 때문에 리팩토링에 강합니다.

어떤 도구를 써도 괜찮습니다. 중요한 것은 도구가 아니라 **어떻게 요청하느냐**입니다. 그게 프롬프트 엔지니어링입니다.

---

### [Ch0-6] 효과적인 프롬프트 4단계 공식

좋은 프롬프트는 4가지 요소로 구성됩니다. **역할 + 컨텍스트 + 구체적 요청 + 제약조건**입니다.

**[예시를 가리키며]** 나쁜 예: "API 연동 코드 만들어줘." 이렇게 하면 AI가 무엇을 만들어야 할지 모릅니다.

좋은 예: "너는 Spring Boot 3.3 백엔드 개발자야. 공공데이터포털 K-water 수질 API를 RestTemplate으로 연동하는 ApiClient 클래스를 만들어줘. 인증키는 환경변수로 관리하고, 응답 JSON을 QualityRecord 리스트로 변환해줘."

차이가 보이시나요? **역할을 먼저 부여하고, 사용 기술 스택을 명시하고, 구체적으로 무엇을 원하는지 말하고, 제약사항을 포함**시켰습니다. 이 공식만 기억해도 AI와의 대화 품질이 3배는 좋아집니다.

### [Ch0-7] 프롬프트 패턴 4가지

실전에서 자주 쓰는 프롬프트 패턴 4가지를 배워봅시다.

**[패턴 1 - 코드 생성]** "너는 [역할]이야. [기술 스택] 환경에서 [기능]을 구현해줘. [제약 조건]을 지켜줘." 이것이 기본 코드 생성 패턴입니다.

**[패턴 2 - 디버깅]** "이 에러가 발생했어: [에러 메시지]. 관련 코드: [코드 붙여넣기]. 원인과 해결 방법을 알려줘." 에러 메시지만 보내는 것보다 관련 코드를 함께 보내면 훨씬 정확한 답을 얻습니다.

**[패턴 3 - 코드 리뷰]** "아래 코드에서 NPE 위험, 보안 이슈, 성능 문제를 찾아줘. 개선 방법도 알려줘." PR 올리기 전에 이 패턴으로 사전 점검하면 됩니다.

**[패턴 4 - 문서화]** "아래 Controller 코드를 보고 API 명세서를 Markdown으로 작성해줘. 요청 파라미터, 응답 JSON 예시를 포함해줘." 코드 완성 후 문서 자동 생성에 활용합니다.

### [Ch0-8] 바이브 코딩 개발 사이클

수자이 프로젝트에서 실제로 사용한 개발 사이클입니다.

**[흐름도를 따라가며]** 먼저 아이디어를 자연어로 적습니다. 그 다음 AI에게 설계 초안을 요청합니다. 초안을 검토하고 방향을 잡은 후 코드 생성을 요청합니다. 생성된 코드를 실행해보고 버그가 있으면 AI에게 에러 메시지를 붙여넣어 해결합니다. 동작이 확인되면 AI에게 문서화를 요청합니다.

이 사이클이 한 번 돌아가는 데 기능 1개 기준으로 **30분~2시간** 정도 걸립니다. 전통 방식으로는 반나절~하루가 걸리는 작업입니다.

### [Ch0-9] 바이브 코딩 주의사항

바이브 코딩이 만능은 아닙니다. 조심해야 할 부분을 짚고 넘어가겠습니다.

**[주의사항 1]** AI가 생성한 코드를 무조건 신뢰하지 마세요. 특히 보안 관련 코드, 인증 처리, API 키 관리 부분은 반드시 직접 검토해야 합니다. AI는 가끔 그럴듯해 보이지만 틀린 코드를 만들기도 합니다. 이것을 "할루시네이션"이라고 부릅니다.

**[주의사항 2]** 저작권과 라이선스를 확인하세요. AI가 생성한 코드가 오픈소스 라이선스와 충돌할 수 있습니다.

**[주의사항 3]** 팀 내 지식 편중을 주의하세요. 한 명만 AI를 잘 다루고 나머지 팀원이 이해하지 못하면 팀 프로젝트가 아닙니다. 모든 팀원이 AI와 대화하고 코드를 이해하는 능력을 키워야 합니다.

### [Ch0-10] 수자이 프로젝트 소개

자, 그럼 오늘 우리가 참고할 프로젝트 수자이(水Xai)를 소개하겠습니다.

**[슬라이드를 보며]** 수자이는 실시간 수질 모니터링 대시보드입니다. K-water 공공 API에서 수질 데이터를 받아오고, Groq AI가 이상 징후를 분석해서 알려주는 시스템입니다. Spring Boot 3.3 백엔드에 Vanilla JS 프론트엔드로 구성된 풀스택 웹 애플리케이션입니다.

이 프로젝트는 바이브 코딩으로 개발했습니다. 설계 문서 작성부터 코드 구현, API 연동, 문서화까지 모든 단계에서 AI를 활용했습니다. 오늘 워크샵에서 수자이를 분해하면서 여러분도 같은 방식으로 프로젝트를 만들어볼 수 있게 됩니다.

### [Ch0-12] 팀 구성과 역할 분담

*[실습 01 발표 후]*

팀 프로젝트에서 역할 분담을 어떻게 해야 할까요?

수자이처럼 4~5명 규모의 소규모 팀에서는 다음과 같이 분담하는 것을 추천합니다. **PM 겸 기획자** 1명은 요구사항을 정리하고 팀을 조율합니다. **백엔드 개발자** 1~2명은 Spring Boot API와 DB를 담당합니다. **프론트엔드 개발자** 1명은 화면을 구현합니다. **문서화 담당** 1명은 설계 문서와 API 명세서를 관리합니다.

중요한 것은 역할이 고정되어 있지 않다는 것입니다. 바이브 코딩 방식에서는 AI가 코드를 생성해주기 때문에 프론트엔드 담당자도 백엔드 코드를 이해하고 수정할 수 있어야 합니다. 오늘 워크샵을 통해 모든 팀원이 전 영역을 경험해볼 수 있습니다.

## Chapter 1. 기획 단계
*소요 시간: 약 90분 | 테마 색상: 딥블루(#065A82)*

---

### [Ch1-1] 챕터 1 표지

챕터 1 기획 단계를 시작합니다.

기획은 프로젝트의 방향을 결정하는 단계입니다. 여기서 결정이 잘못되면 나중에 아무리 코드를 잘 짜도 방향이 틀린 것이 됩니다. 반대로 기획이 잘 되어 있으면 구현은 훨씬 쉬워집니다.

AI를 활용하면 기획 단계도 훨씬 빠르게 진행할 수 있습니다. 아이디어를 AI와 대화하면서 구체화하고, 요구사항 초안을 자동으로 생성하는 방법을 배워보겠습니다.

---

### [Ch1-2] 이번 챕터 학습 목표

이번 챕터에서 배울 내용입니다.

첫째, 아이디어를 서비스 요구사항으로 변환하는 방법을 배웁니다. 둘째, 기능 요구사항(FR)과 비기능 요구사항(NFR)을 구분하는 법을 배웁니다. 셋째, 이해관계자 분석을 통해 누구를 위한 서비스인지 명확히 합니다. 넷째, In/Out Scope를 결정해서 MVP 범위를 정의합니다. 다섯째, AI로 요구사항 정의서를 작성하는 방법을 실습합니다.

예상 소요시간은 90분입니다. 실습이 포함되어 있으니 적극적으로 참여해주세요.

---

### [Ch1-3] 왜 기획이 먼저인가?

"코딩부터 시작하면 안 되나요?" 이런 질문을 많이 받습니다.

**[슬라이드를 보며]** 기획 없이 시작하면 어떻게 될까요? 개발하다 보면 "이 기능도 있으면 좋겠는데"가 계속 추가됩니다. 범위가 끝없이 늘어나고, 팀원마다 다른 그림을 그리게 됩니다. 이것을 스코프 크리프(Scope Creep)라고 합니다. 결국 마감일을 못 지키거나, 완성도가 낮은 채로 제출하게 됩니다.

**[오른쪽을 가리키며]** 기획이 먼저 되어 있으면 팀 전체가 같은 방향을 바라봅니다. "이건 우리 범위야/아니야"를 빠르게 판단할 수 있고, 구현 속도도 훨씬 빨라집니다. 수자이에서 기획 문서가 완성된 후 구현 속도가 3배 빨라졌습니다.

---

### [Ch1-4] 아이디어를 서비스로 만들기

아이디어에서 서비스로 가는 과정을 살펴봅시다.

수자이의 경우, 아이디어는 이랬습니다. "수질 데이터를 쉽게 볼 수 있으면 좋겠다." 이것을 AI와 대화하며 구체화했습니다.

**AI에게 이렇게 물었습니다**: "실시간 수질 모니터링 대시보드를 만들려고 해. 공공 API를 활용할 건데, 어떤 기능이 있으면 좋을까?" AI가 기능 목록을 제안해주면, 팀이 검토해서 "이건 필요해, 이건 나중에 하자"를 결정합니다.

이 과정이 **이해관계자 분석**과 연결됩니다. 수자이의 이해관계자는 수질 데이터를 모니터링하는 운영자, 이상 징후를 보고받는 관리자, API를 제공하는 K-water입니다. 각 이해관계자가 무엇을 필요로 하는지를 파악하면 요구사항이 자연스럽게 도출됩니다.

### [Ch1-5] 이해관계자 분석 — 누구를 위한 서비스인가?

서비스를 만들기 전에 반드시 물어야 할 질문이 있습니다. "이 서비스를 누가 쓸 것인가? 그들이 원하는 것이 무엇인가?"

**[이해관계자 목록을 보며]** 수자이의 경우 세 그룹의 이해관계자가 있습니다.

첫 번째, **수질 모니터링 운영자**: 실시간으로 여러 정수장의 수질 현황을 한눈에 보고 싶습니다. 수치만 나열되는 것보다 정상/주의/위험 같은 상태 표시가 필요합니다.

두 번째, **관리자**: 이상 징후가 발생했을 때 즉각 알림을 받고, AI 분석을 통해 원인과 대응 방안을 빠르게 파악하고 싶습니다.

세 번째, **K-water API 제공자**: 우리가 API를 남용하지 않도록 적절한 캐싱이 필요합니다.

여러분의 프로젝트에서도 이해관계자 분석을 먼저 해보세요. 그러면 기능 목록이 자연스럽게 나옵니다.

### [Ch1-6] 기능 요구사항(FR) 정의

이해관계자 분석이 끝나면 기능 요구사항을 정의합니다.

**[FR 목록을 보며]** 수자이의 기능 요구사항은 6가지입니다.

- FR-001: 정수장 목록 조회 — 전국 정수장 목록을 드롭다운으로 선택
- FR-002: 수질 데이터 조회 — 선택한 정수장의 수질 항목별 수치 표시
- FR-003: 유량 데이터 조회 — 유입량, 방류량 등 유량 정보 표시
- FR-004: AI 수질 분석 — Groq AI가 이상 징후 분석 결과 제공
- FR-005: 수질 경보 표시 — 기준 초과 시 색상 경보 (정상/주의/위험)
- FR-006: 캐시 최적화 — 동일 요청 반복 시 API 호출 없이 캐시 반환

**AI에게 이런 식으로 요청합니다**: "수질 모니터링 대시보드를 만들 건데, 필요한 기능 요구사항(FR)을 FR-번호 형식으로 목록화해줘. 각 요구사항에 설명도 포함해줘." AI가 초안을 만들어주면 팀이 추가하거나 삭제합니다.

### [Ch1-7] 비기능 요구사항(NFR) — 품질 기준 정의

기능 요구사항이 "무엇을 할 것인가"라면, 비기능 요구사항은 "어떻게 잘 할 것인가"입니다.

**[NFR 카테고리를 보며]** 수자이의 비기능 요구사항은 4가지 카테고리입니다.

- NFR-001 **성능**: 캐시 히트 시 200ms 이내 응답. 이게 없으면 API를 매번 호출해서 수십 초가 걸립니다.
- NFR-002 **가용성**: 외부 API 장애 시 캐시 데이터로 부분 서비스 유지
- NFR-003 **보안**: API 키 환경변수 관리, .gitignore로 키 노출 방지
- NFR-004 **유지보수성**: ApiClient 1:1 대응 패턴으로 코드 구조 명확화

NFR은 나중에 코드 품질 기준이 됩니다. "이 API 응답이 200ms가 넘는데 어떻게 해결하지?"라는 상황이 올 때 NFR이 판단 기준이 됩니다.

### [Ch1-8] In Scope / Out Scope — MVP 경계선 긋기

프로젝트에서 가장 중요한 결정 중 하나는 **무엇을 하지 않을지 결정하는 것**입니다.

**[In Scope를 보며]** 수자이의 첫 번째 버전에 포함한 것들입니다. 수질/유량 데이터 조회, AI 분석, 기본 경보 색상 표시. 이것만으로도 핵심 가치를 전달할 수 있습니다.

**[Out Scope를 보며]** 하지 않기로 결정한 것들입니다. 실시간 알림 시스템, 사용자 로그인, 히스토리 저장, 모바일 앱. 이것들이 "있으면 좋겠다"이지만 첫 버전에 다 넣으려고 하면 아무것도 완성하지 못합니다.

AI에게도 In/Out Scope 정리를 요청할 수 있습니다. "우리 서비스의 MVP 범위를 In Scope와 Out Scope로 나눠줘. Phase 2, Phase 3 로드맵도 작성해줘." 이렇게 하면 팀이 방향을 맞추는 데 도움이 됩니다.

- UC-003: 유량 데이터 조회 — 유입량, 방류량 등 유량 정보 표시

### [Ch2-5] 아키텍처 선택 — 왜 이 구조인가?

수자이는 **3레이어 MVC 아키텍처**를 선택했습니다. 왜 이 구조인가?

**[레이어 구조를 보며]** 크게 3개 레이어입니다.

- **프레젠테이션 레이어**: Controller와 index.html. 사용자 요청을 받고 응답을 반환합니다.
- **비즈니스 레이어**: Service. 캐시 로직과 비즈니스 규칙을 처리합니다.
- **데이터 레이어**: ApiClient. 외부 API를 호출합니다.

이 구조를 선택한 이유가 중요합니다. Spring Boot의 표준 패턴이라 AI가 이 구조를 완벽하게 이해합니다. AI에게 "Spring Boot MVC로 이 기능 만들어줘"라고 하면 바로 이 구조로 코드를 생성합니다. 팀원 모두 같은 구조를 이해하면 협업이 쉬워집니다.

### [Ch2-6] 레이어 구조와 클래스 설계

각 레이어의 클래스를 구체적으로 설계합니다.

**[클래스 목록을 보며]** 수자이의 주요 클래스입니다.

Controller: `ApiController` — 5개 REST API 엔드포인트 담당

Service: `WaterQualityService`, `WaterFluxService`, `FacilityService`, `GroqService` — 각각 캐시 로직 포함

ApiClient: `WaterQualityApiClient`, `WaterFluxApiClient`, `FacilityApiClient`, `GroqApiClient` — 외부 API 1:1 대응

중요한 패턴은 **Service ↔ ApiClient 1:1 대응**입니다. 예를 들어 `WaterQualityService`는 `WaterQualityApiClient`만 호출합니다. 이 패턴을 AI에게 설명하면 이후 모든 코드 생성에서 이 패턴을 따릅니다.

### [Ch2-7] 기술 스택 결정 — AI에게 추천받기

기술 스택 선택도 AI의 도움을 받을 수 있습니다.

수자이에서 AI에게 이렇게 물었습니다: "수질 데이터 모니터링 백엔드를 만들려고 해. 공공 API 연동, 캐시, AI API 연동이 필요한데 적합한 기술 스택을 추천해줘. 팀원이 Java에 익숙해."

AI가 추천한 스택이 실제로 수자이에 적용된 것들입니다.

- **Spring Boot 3.3**: Java 표준 백엔드 프레임워크, Spring Security 미사용 (MVP 단계 단순화)
- **RestTemplate**: 외부 API 호출. WebClient보다 학습 곡선이 완만
- **ConcurrentHashMap**: 인메모리 캐시. Redis 없이도 충분 (MVP 단계)
- **Groq AI (LLaMA-3.3-70b)**: 무료 티어 제공, OpenAI 호환 형식
- **Vanilla JS**: 프레임워크 학습 비용 없음, 단일 파일 배포

**[팁]** AI에게 기술 스택을 추천받을 때 "팀의 현재 기술 수준"을 함께 알려주세요. 더 현실적인 추천을 받을 수 있습니다.

### [Ch2-9] 캐시 전략 설계

캐시는 성능과 비용 절감의 핵심입니다. 수자이의 캐시 전략을 설명하겠습니다.

**[TTL 설계를 보며]** 데이터 종류에 따라 다른 TTL을 적용했습니다.
- 수질/유량 데이터: **10분** TTL — 실시간성이 중요하지만 매 요청마다 API 호출은 낭비
- Groq AI 분석: **5분** TTL — 무료 티어 토큰 한도 보호
- 정수장 목록: **24시간** TTL — 거의 변하지 않는 데이터

**캐시 키 설계**: `sujCode|startDate|endDate` 형식입니다. 같은 정수장이라도 날짜가 다르면 다른 캐시 항목으로 처리됩니다.

AI에게 "ConcurrentHashMap으로 TTL 기반 인메모리 캐시를 구현해줘. 캐시 키는 sujCode+날짜 조합이고 TTL은 10분이야"라고 요청하면 전체 구현 코드가 나옵니다.

### [Ch3-5] 공공 API 연동 — 핵심 포인트

공공 API, 특히 K-water API를 연동할 때 주의해야 할 4가지 포인트가 있습니다. 이것은 수자이 개발에서 가장 많은 시간을 투자한 부분입니다.

**[첫 번째 - 중첩 JSON 파싱]** K-water API 응답은 `response.body.items.item` 이렇게 4단계로 중첩되어 있습니다. 일반적인 API보다 복잡합니다.

**[두 번째 - 단건 vs 복수 불일치]** 결과가 1건이면 `Map`으로, 여러 건이면 `List<Map>`으로 옵니다. 이것이 `ClassCastException`의 원인입니다. `instanceof` 체크가 필수입니다.

**[세 번째 - items 빈 문자열]** 데이터가 없을 때 `items`가 `null`이 아니라 빈 문자열 `""`로 옵니다. `null` 체크만 하면 NPE가 발생합니다.

**[네 번째 - sujCode 미입력 시 44초]** 정수장 코드 없이 호출하면 전체 데이터를 조회해서 44초 지연이 발생합니다. `sujCode`를 필수 파라미터로 처리해야 합니다.

이 4가지를 AI에게 미리 알려주면 이 모든 처리를 포함한 코드를 생성해줍니다.

### [Ch3-6] AI에게 JSON 붙여넣기 → 파싱 코드 자동 생성

이 방법이 수자이 개발에서 가장 시간을 절약한 AI 활용 사례입니다. 실제 응답 JSON이 있으면 AI가 정확한 코드를 만들어줍니다.

### [Ch3-8] AI 분석 품질을 높이는 프롬프트 설계

**[프롬프트 반복 개선 과정을 보며]** 수자이에서 실제로 거친 과정입니다.

### [Ch3-11] 버그가 생겼을 때 AI와 함께 해결하기

예상 소요시간은 60분입니다.

### [Ch4-3] 문서화 없이 vs AI로 문서화한 팀

6개월 후를 생각해봅시다.

**[A팀 - 문서 없이 개발]** 코드만 있고 문서가 없는 팀의 6개월 후입니다. 코드 수정이 불가능합니다. 팀원이 이직하면 모든 지식이 사라집니다. 신규 팀원 온보딩에 2주가 걸립니다. API가 바뀌면 어디를 수정해야 할지 모릅니다. 프론트-백 협업 때마다 구두로 설명해야 합니다.

**[B팀 - AI로 문서화 병행]** 언제든 코드 수정이 가능합니다. API 명세서로 프론트-백 협업이 원활합니다. 신규 팀원 온보딩이 1일 만에 완료됩니다. 아키텍처 문서로 변경 영향을 즉시 파악합니다. PR 리뷰로 코드 품질이 자동 점검됩니다.

AI 덕분에 문서 초안 작성 시간이 대폭 줄었습니다. 문서화가 더 이상 귀찮은 일이 아닙니다.

### [Ch4-4] 수자이 프로젝트의 5개 설계 문서

수자이에서 작성한 5가지 설계 문서를 소개합니다.

**[표를 보며]** 각 문서와 AI 역할입니다.

1. **요구사항 정의서**: AI가 사용자 스토리 초안 작성 → 약 2시간
2. **유스케이스 정의서**: AI가 UC 목록 도출 + 예외 흐름 생성 → 약 1.5시간
3. **시스템 아키텍처 정의서**: AI가 레이어 구조 및 기술 스택 제안 → 약 2시간
4. **화면 설계서**: AI가 ASCII 와이어프레임 생성 → 약 1.5시간
5. **API 명세서**: 코드 → AI → 문서 자동화 → 약 1시간

AI 없이 약 2주 소요 → AI 활용 시 약 2일 완성. 모든 문서에서 AI가 초안을 잡고 팀이 검토·보완하는 방식으로 진행했습니다.

### [Ch4-5] 코드 → AI → API 명세서 자동 생성

API 명세서 자동 생성은 가장 실용적인 AI 문서화 활용 사례입니다.

**[흐름을 따라가며]** Controller 코드 완성 → AI에게 붙여넣기 → Markdown 명세서 생성 → 팀 검토 & 배포

AI에게 이렇게 요청합니다: "아래 Spring Boot Controller를 보고 API 명세서를 Markdown으로 작성해줘. 요청 파라미터, 응답 필드, JSON 예시를 포함해줘."

수자이의 5개 내부 API가 이 방법으로 30분 만에 명세서가 완성됐습니다. 생성된 명세서는 Notion이나 GitHub Wiki에 바로 붙여넣을 수 있습니다. 팀원이 API를 사용할 때 구두로 설명할 필요가 없어집니다.

**[API 목록을 보며]** 수자이의 5개 REST API입니다.
- `GET /api/facilities` — 전체 정수장 목록 조회
- `GET /api/waterQuality` — 수질 데이터 조회
- `GET /api/waterFlow` — 유량 데이터 조회
- `GET /api/analysis` — Groq AI 수질 분석
- `GET /api/news` — 수질 관련 뉴스 검색

### [Ch4-6] 외부 API 연동 명세 — AI로 정리하기

외부 API 연동 명세서도 AI로 정리할 수 있습니다.

**[테이블을 보며]** 수자이의 외부 API 4개 명세입니다.

- **K-water 수질 API**: `apis.data.go.kr/1490000/wtrQltService` — items 빈 문자열 처리, occrrncDt 날짜 변환 주의
- **K-water 유량 API**: `apis.data.go.kr/1490000/waterFlowService` — 단건/복수 응답 불일치, instanceof Map 체크 필요
- **K-water 정수장 API**: 파라미터 없음 (전체 조회) — 60분 캐시로 빈번 호출 방지
- **Groq AI API**: `api.groq.com/openai/v1/chat/completions` — OpenAI 호환, 한국어 전용 시스템 프롬프트

특수 처리 항목들, 예를 들어 날짜 변환 형식이나 빈 문자열 처리 방법도 명세서에 포함시키세요. 이런 예외 상황이 문서화되어 있지 않으면 나중에 팀원이 같은 문제를 다시 겪게 됩니다.

### [Ch4-7] 시스템 아키텍처 정의서 — 설계 결정 근거(Why)까지

아키텍처 정의서에서 가장 중요한 섹션은 **설계 결정 사항(Why)**입니다.

**[Why 섹션을 보며]** AI는 What과 How는 잘 작성해주지만 Why는 팀이 직접 보완해야 합니다.

수자이의 설계 결정 근거들입니다.
- **DB 미사용**: MVP 단계에서 비용 최소화, ConcurrentHashMap으로 충분
- **sujCode 필수화**: 44초 지연 방지, API 안정성 확보
- **Groq 선택**: 무료 티어 즉시 사용, OpenAI 호환으로 코드 재사용 가능
- **Vanilla JS**: 프레임워크 학습 비용 없음, 단일 파일 배포 가능
- **Maven 선택**: 팀원 IDE 환경 통일, 의존성 관리 표준화

Why 섹션이 6개월 후 가장 큰 가치를 발휘합니다. "왜 DB를 쓰지 않았지?"라는 질문에 바로 답할 수 있게 됩니다.

⚠ AI가 Why를 놓치는 경우가 많습니다. 팀이 직접 보완해야 하는 부분입니다.

### [Ch4-8] 팀 협업 도구와 Git 전략

팀 협업을 위한 Git 전략과 AI 활용법입니다.

**[브랜칭 전략을 보며]** main → develop → feature/xxx 3단계 브랜칭이 소규모 팀에 적합합니다.
- `main`: 배포 가능 버전만 (절대 직접 push 금지)
- `develop`: 통합 개발 브랜치
- `feature/xxx`: 기능별 개발 브랜치 (팀원당 1개)

**PR 작성 with AI**: AI에게 코드 변경사항을 붙여넣으며 "이 변경사항으로 PR 설명을 작성해줘. 제목, 변경 내용 요약, 테스트 방법을 포함해줘"라고 하면 됩니다.

**코드 리뷰 with AI**: 팀 리뷰 전에 AI에게 먼저 점검합니다.
- "이 코드에서 잠재적 버그와 NPE 위험을 찾아줘"
- "API 키 노출, 인증 미흡 등 보안 이슈를 확인해줘"
- "이 함수에 대한 JUnit 테스트 케이스를 작성해줘"

AI PR 리뷰는 팀 리뷰 전 사전 점검으로 활용해서 명백한 버그를 미리 걸러내세요.

### [Ch4-10] AI 협업 개발 전체 워크플로우

워크샵 전체를 하나의 워크플로우로 정리하겠습니다.

**[전체 흐름을 따라가며]**

- **아이디어**: AI — 유사 서비스 분석, 아이디어 정제 / 팀 — 문제 정의, 목표 설정
- **기획**: AI — 요구사항 초안, 사용자 스토리 / 팀 — 범위 결정, 우선순위
- **설계**: AI — 아키텍처 제안, UC 목록 생성 / 팀 — 기술 결정, 팀 합의
- **구현**: AI — 코드 생성, 버그 디버깅 / 팀 — 코드 검토, 통합 테스트
- **문서화**: AI — 명세서 자동생성, PR 설명 작성 / 팀 — Why 보완, 리뷰 승인

핵심 메시지를 기억하세요. **AI는 초안 작성과 코드 생성을, 팀은 검토와 결정을 담당합니다.**

전통 방식: 아이디어 → 출시 평균 6~8주 / AI 협업 방식: 아이디어 → 출시 평균 1~2주. 이 속도 차이가 바이브 코딩의 힘입니다.

### Q&A 예상 질문

**Q: AI가 틀린 코드를 생성하면 어떻게 하나요?**
A: 에러 메시지를 복사해서 AI에게 다시 보내세요. "이 에러가 발생했어. 원인과 해결 방법을 알려줘"라고 하면 됩니다. 보통 2~3번 대화로 해결됩니다.

**Q: 어떤 AI를 써야 하나요?**
A: 무엇을 써도 괜찮습니다. Claude, ChatGPT, Cursor 모두 비슷한 수준의 코드를 생성합니다. 중요한 것은 도구가 아니라 프롬프트 작성 능력입니다.

**Q: AI를 쓰면 실력이 늘지 않는 것 아닌가요?**
A: AI가 만든 코드를 이해하고 검토하는 과정에서 실력이 향상됩니다. 모르는 코드를 AI에게 설명해달라고 하는 것도 훌륭한 학습 방법입니다.

**Q: 공공 API 인증키 발급이 안 될 때는?**
A: 강사가 준비한 샘플 JSON 응답 파일을 사용하면 됩니다. 실습은 JSON 파싱 연습이 핵심이므로 실제 API 없이도 진행 가능합니다.
