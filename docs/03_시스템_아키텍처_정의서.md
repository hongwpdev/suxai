## 4. 데이터 흐름

### 4.1 정수장 목록 조회 흐름
```
Browser
  → GET /api/facilities
  → FacilityService (캐시 확인)
    → [캐시 유효] 캐시 데이터 반환
    → [캐시 만료] FacilityApiClient
      → K-water 정수장 목록 API (HTTPS GET)
      → JSON 파싱 → List<FacilityInfo>
      → 캐시 저장 (TTL: 24시간)
  ← 41개 정수장 목록 (JSON)
```

### 4.2 수질 데이터 조회 흐름
```
Browser
  → GET /api/waterQuality?sujCode=318
  → QualityService (sujCode별 캐시 확인)
    → [캐시 유효] 캐시 데이터 반환
    → [캐시 만료] QualityApiClient
      → K-water 수질 API (HTTPS GET, 최근 1시간)
      → JSON 파싱 → List<WaterQualityRecord>
      → 캐시 저장 (TTL: 5분)
  ← 수질 레코드 목록 (JSON)
```

### 4.3 AI 분석 흐름
```
Browser
  → GET /api/analysis?sujCode=318
  → AnalysisService (sujCode별 캐시 확인)
    → [캐시 유효] 캐시 결과 반환
    → [캐시 만료]
      → QualityService.getRecords(sujCode)
      → FlowService.getRecords(sujCode)
      → 두 데이터를 fcltyMngNo 기준으로 조인
      → 프롬프트 생성 (수질·유량 통합 / 수질만 / 유량만)
      → GroqApiClient → Groq API (HTTPS POST)
      → 한국어 분석 리포트 수신
      → 캐시 저장 (TTL: 5분)
  ← AnalysisResult (JSON)
```

## 5. 캐시 전략

| 서비스 | 캐시 키 | TTL | 저장소 |
|--------|---------|-----|--------|
| FacilityService | 단일 (전체 목록) | 24시간 | volatile 필드 |
| QualityService | sujCode | 5분 | ConcurrentHashMap |
| FlowService | sujCode | 5분 | ConcurrentHashMap |
| AnalysisService | sujCode | 5분 | ConcurrentHashMap |

**캐시 갱신 정책**: 요청 시점에 TTL 만료 여부를 확인하여 갱신 (Pull-through Cache)

## 7. 주요 설계 결정 사항

| 결정 | 이유 |
|------|------|
| DB 미사용 (인메모리 캐시) | MVP 단계에서 실시간 조회만 필요, DB 구축 비용 최소화 |
| sujCode 파라미터 필수화 | sujCode 없이 조회 시 API 응답 시간 44초 이상 소요 |
| Groq 선택 (vs Gemini) | 무료 티어 즉시 사용 가능, OpenAI 호환 API |
| Vanilla JS 선택 | 프레임워크 없이 단일 HTML 파일로 서빙, 배포 간소화 |
| 한글 전용 AI 출력 | 시스템 프롬프트로 한자·영어 출력 금지 적용 |
| XxxService ↔ XxxApiClient 네이밍 | 서비스-클라이언트 1:1 대응 명확화, 가독성 향상 |
