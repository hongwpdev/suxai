### ISSUE-001. K-water API 응답 지연 (44초 이상)

**현상**

K-water 수질 API를 `sujCode` 파라미터 없이 호출하면 응답이 44초 이상 소요되어 사실상 서비스 불가 상태가 됨.

**원인**

`sujCode` 미입력 시 전국 전체 정수장 데이터를 조회하기 때문에 응답 시간이 크게 늘어남. 공공데이터 API 문서에는 해당 파라미터가 선택(Optional)으로 표기되어 있어 초기에는 인지하지 못했음.

**해결 방법**

`sujCode`를 요청 파라미터로 필수 지정하고, `application.properties`에 기본값(파주정수장 코드 `318`)을 설정하여 미입력 시에도 기본 정수장으로 조회되도록 처리함.

```
# application.properties
kwater.suj.code = 318
```

아울러 `AppConfig.java`에서 RestTemplate 타임아웃을 실측값 기준으로 설정함.

```java
// AppConfig.java
factory.setConnectTimeout(5_000);   // 연결 타임아웃 5초
factory.setReadTimeout(25_000);     // 읽기 타임아웃 25초
```

**교훈**

공공 API의 선택 파라미터라도 직접 호출 테스트를 통해 응답 시간을 먼저 측정해볼 것. 타임아웃 값은 임의로 설정하지 말고 실측 기반으로 결정할 것.

### ISSUE-002. AI 응답에 한자 혼입

**현상**

Groq AI(LLaMA 3.3 70B)에 수질 데이터 분석을 요청하면 "殺菌 능력", "濃度 기준" 등 한자가 섞인 응답이 반환됨. 일반 사용자 대상 서비스에서 가독성 문제 발생.

**원인**

LLaMA 모델이 학습 데이터 특성상 한국어 전문 용어를 표기할 때 한자를 혼용하는 경향이 있음. 별도 지시가 없으면 모델이 자체 판단으로 한자를 사용함.

**해결 방법**

시스템 프롬프트에 한글 전용 출력 및 한자 금지를 명시함.

```java
// GroqApiClient.java
Map.of("role", "system", "content",
    "당신은 수처리 전문가입니다. " +
    "반드시 한국어(한글)로만 답변하세요. " +
    "한자(殺菌, 濃度 등 한문)를 절대 사용하지 마세요. " +
    "한자 대신 반드시 한글을 사용하세요: 살균, 농도, 능력, 부족 등. " +
    "영어 단어도 가능하면 한국어로 바꾸세요.")
```

**교훈**

LLM을 사용할 때 출력 언어, 금지 표현, 역할을 시스템 프롬프트에 명확히 지정해야 함. "한국어로 답변하세요"만으로는 부족하고, 금지 사례를 구체적으로 예시로 제시해야 효과적.

### ISSUE-003. 공공 API 응답 형식과 문서 불일치

**현상**

K-water API 공식 문서에 명시된 JSON 응답 필드명과 실제 응답 데이터의 필드명이 일부 다르거나, 단건 응답 시 배열이 아닌 객체로 반환되는 경우 발생.

**원인**

공공 API 특성상 문서 업데이트가 실제 API 변경을 즉시 반영하지 않는 경우가 있음. 또한 조회 결과가 1건일 때 `item`이 배열(`[]`)이 아닌 단일 객체(`{}`)로 내려오는 구조적 특성이 있음.

**해결 방법**

응답 파싱 시 `item`이 배열인지 단일 객체인지 분기 처리함.

```java
// QualityApiClient.java
Object itemObj = itemsMap.get("item");

if (itemObj instanceof List<?> list) return (List<Map<String, Object>>) list;
if (itemObj instanceof Map)           return List.of((Map<String, Object>) itemObj);
return Collections.emptyList();
```

**교훈**

공공 API를 연동할 때는 문서를 참고하되 반드시 직접 호출 테스트를 먼저 진행할 것. 특히 단건/다건 응답 형식이 다른 경우를 반드시 방어 처리해야 함.

### TIP-001. 프로토타입 먼저, 전환은 나중에

**내용**

Spring Boot 개발에 앞서 Python(Streamlit)으로 프로토타입을 먼저 제작했음. `app.py`에서 UI 레이아웃과 카드 렌더링을, `data.py`에서 이상징후 점수 계산 로직을, `utils/anomaly.py`에서 임계값 판정 로직을 빠르게 구현하고 검증함.

**장점**

- Spring Boot 개발 전에 화면 구성과 비즈니스 로직을 미리 확인할 수 있음
- `jitter()` 함수로 실시간 노이즈를 시뮬레이션하여 실 API 없이도 전체 흐름을 테스트 가능

```python
# data.py
def jitter(value, amp, lo, hi):
    return round(float(np.clip(value + np.random.uniform(-amp, amp), lo, hi)), 2)
```

**적용 팁**

프로토타입에서 확인한 이상징후 점수 계산 로직(`wq_score`, `fl_score`)과 임계값 기준은 Spring Boot로 전환 시 AI 프롬프트의 기준값으로 그대로 활용할 수 있음.

### TIP-002. AI API 호출 비용과 지연은 캐시로 해결

**내용**

Groq AI API 호출은 응답까지 수 초가 소요되고, 무료 티어는 분당 토큰 제한이 있음. 동일한 정수장에 대해 여러 사용자가 동시에 분석을 요청하면 중복 호출이 발생함.

**해결 방법**

`sujCode`(정수장 코드)를 키로 하는 `ConcurrentHashMap` 캐시를 적용하여 5분 이내 동일 정수장 재요청 시 AI를 다시 호출하지 않고 캐시 결과를 반환함.

```java
// AnalysisService.java
Instant exp = expiry.get(sujCode);
if (exp != null && Instant.now().isBefore(exp)) {
    return cache.get(sujCode);  // AI 재호출 없이 즉시 반환
}
```

| 서비스 | 캐시 키 | TTL |
|--------|---------|-----|
| 정수장 목록 | 단일 | 24시간 |
| 수질 데이터 | sujCode | 5분 |
| 유량 데이터 | sujCode | 5분 |
| AI 분석 결과 | sujCode | 5분 |

**적용 팁**

데이터 성격에 따라 TTL을 다르게 설정하는 것이 핵심. 변화가 거의 없는 정수장 목록은 24시간, 실시간성이 중요한 수질·유량·AI 분석은 5분으로 구분함.

### TIP-003. 프롬프트 설계가 AI 출력 품질을 결정한다

**내용**

단순히 데이터를 나열해서 AI에 전달하면 분석 품질이 낮고 출력 형식이 일관되지 않음. 프롬프트를 구조화하면 품질이 크게 향상됨.

**적용 방법**

① 데이터를 의미 단위로 섹션 분리 (수질+유량 통합 / 수질만 / 유량만)

```java
// AnalysisService.java
// fcltyMngNo 기준으로 조인 후 3섹션으로 분리
Set<String> joinedIds  = ...;  // 상관관계 분석 가능
Set<String> wqOnlyIds  = ...;  // 수질 단독 분석
Set<String> flOnlyIds  = ...;  // 유량 단독 분석
```

② 출력 형식과 항목을 프롬프트에 명시

```
**1. 전체 현황**  ← 2~3줄 요약
**2. 이상징후**   ← 기준값 포함 명시 (pH 6.5~8.5, 탁도 0.5 NTU 이하...)
**3. 권고사항**   ← 없으면 "특이사항 없음" 으로 답변
```

③ 결과 없음 케이스의 답변 문구도 지정

**적용 팁**

출력 형식을 마크다운 볼드(`**항목**`)로 지정하면 프론트엔드에서 별도 파싱 없이 `<b>` 태그로 변환해 렌더링 가능.

### TIP-004. API 키는 처음부터 분리해서 관리할 것

**내용**

개발 초기에 API 키를 소스코드에 직접 작성하면 실수로 GitHub에 올라갈 위험이 있음. 처음부터 분리해서 관리하는 습관이 중요함.

**적용 방법**

```
application.properties         ← 실제 키 포함, .gitignore 처리
application.properties.example ← 키 형식(플레이스홀더)만 기재, GitHub 공개
```

```
# application.properties.example
kwater.api.key = YOUR_KWATER_API_KEY_HERE
groq.api.key   = YOUR_GROQ_API_KEY_HERE
```

**적용 팁**

`.gitignore`에 `application.properties`를 추가하는 것을 프로젝트 초기화 단계에서 바로 처리할 것. `.example` 파일을 함께 커밋해두면 팀원이 키 설정 방법을 바로 알 수 있음.
